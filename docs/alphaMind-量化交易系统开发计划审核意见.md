# alphaMind 量化交易系统开发计划审核意见

## 1. 审核目的

本审核意见针对现有 `alphaMind` 量化交易系统开发计划，结合项目当前新增目标进行评估：

> 由大语言模型生成候选交易计划，经确定性风险模块校验后，通过 Telegram Bot 发送给用户，由用户批准或拒绝，最终仍由 Freqtrade 作为唯一交易写入者执行。

本审核重点关注：

- 原开发计划与新增 AI 决策目标之间的一致性；
- LLM 决策链的权限边界；
- Telegram 人工审批流程的安全性；
- 模型版本、Prompt、输入输出及审计治理；
- Paper、Live Canary 和反事实评估机制；
- 当前进度账本、Holdout 与独立评审问题。

---

## 2. 总体评价

现有开发计划的传统量化工程部分整体成熟，尤其在以下方面表现较好：

- Freqtrade 保持唯一交易写路径；
- Runtime DB、Audit DB 和 RiskSnapshot 边界清晰；
- 已建立 Fail-Closed、Kill Switch、Close-Only 等风险状态；
- 数据、回测、Paper、Replay 和 Live Canary 分层明确；
- 强调不可变数据、实验登记、版本锁定和审计证据；
- 禁止通过第二套订单系统绕过 Freqtrade；
- 对真实资金上线设置较严格的阶段门禁。

但项目目标已经发生实质变化。原计划明确将 LLM 实时信号和 AI 资金决策排除在 MVP 之外，而当前目标要求 LLM 生成候选交易计划。因此，AI 模块不能作为外围附加功能直接接入，必须作为新的正式决策层重新纳入架构、验证和治理体系。

总体结论：

> 原计划的资金安全与传统量化基础可以保留，但必须重新打开设计冻结阶段，补充 AI 计划生成、人工审批、模型治理和独立评估合同。否则，AI 模块将成为整套系统中唯一缺乏严格约束的部分。

---

## 3. 关键审核发现

### 3.1 原计划与新增目标存在根本冲突

原计划中的以下方向与当前目标不一致：

- AI 不进入实时资金决策链；
- LLM 实时信号不属于 MVP；
- AI Agent 不是第一版前置条件。

当前目标则是：

1. LLM 根据行情、指标和账户状态生成候选交易计划；
2. 程序校验计划格式与风险；
3. Telegram 将计划发送给用户；
4. 用户批准或拒绝；
5. 批准后重新校验；
6. Freqtrade 执行订单。

这属于策略权威与信号来源的实质变化，不能视为普通通知功能。

### 3.2 当前唯一交易写路径应继续保留

建议继续冻结以下原则：

> 只有 Freqtrade 可以创建、修改或取消真实交易订单。

LLM、Telegram Bot、Watchdog、Audit Writer 均不得直接持有交易写权限。

建议采用以下链路：

```text
已完成的 4h K 线
    ↓
确定性行情与指标计算
    ↓
LLM 生成候选交易计划
    ↓
JSON Schema 校验
    ↓
确定性风险与市场规则校验
    ↓
Telegram 待审批
    ↓
用户批准 / 拒绝 / 超时
    ↓
执行前重新校验
    ↓
Freqtrade 唯一写入订单
```

## 4. 必须修改项

### 4.1 新增 AI Proposal Authority 决策合同

建议新增正式任务：

`P0-09 AI Proposal Authority`

该任务必须冻结以下内容：

- 模型供应商；
- 模型名称；
- 模型版本或版本标识；
- API 接口版本；
- System Prompt；
- Prompt Template；
- Prompt Hash；
- 输入字段；
- 输入数据的时间点；
- 输出 JSON Schema；
- 最大输入 Token；
- 最大输出 Token；
- Temperature、Top-P 等采样参数；
- 超时时间；
- 最大重试次数；
- 限频策略；
- API 余额不足处理；
- 模型不可用时的处理；
- 是否允许切换备用模型；
- 模型升级后是否重启 Paper 周期；
- 模型输出审计和留存规则。

推荐原则：

> 模型不可用、输出损坏、版本未知或请求超时时，默认不生成交易计划，不允许自动退化为未验证模型继续决策。

DeepSeek、OpenAI 或其他模型之间的切换，应视为策略版本变化，而非普通基础设施切换。

### 4.2 建立结构化交易计划 Schema

LLM 不应输出一段由程序自由解析的自然语言，而应返回严格结构化数据。

建议示例：

```json
{
  "schema_version": "1",
  "plan_id": "20260717-BTC-001",
  "symbol": "BTC/USDT",
  "decision": "BUY",
  "timeframe": "4h",
  "generated_at": "2026-07-17T08:00:00Z",
  "valid_until": "2026-07-17T08:15:00Z",
  "entry_min": "65900",
  "entry_max": "66300",
  "invalidation_price": "64500",
  "take_profit_prices": ["69000", "71500"],
  "max_holding_hours": 168,
  "reason_codes": [
    "DONCHIAN_BREAKOUT",
    "TREND_CONFIRMATION"
  ],
  "risk_flags": [],
  "confidence": "0.68"
}
```

必须校验：

- 交易对在白名单内；
- 方向只允许 `BUY`、`HOLD`、`EXIT`；
- 所有价格必须为正数；
- 止损与入场方向关系合理；
- 有效期不得超过允许范围；
- 时间戳必须为 UTC；
- 计划不能引用未来数据；
- 计划只能基于已完成 K 线；
- 模型不得自由指定最终仓位；
- 未知字段或 Schema 版本直接拒绝。

### 4.3 Telegram 审批必须实现正式状态机

建议固定计划状态：

```text
CREATED
→ SCHEMA_VALIDATED
→ RISK_VALIDATED
→ PENDING_APPROVAL
→ APPROVED / REJECTED / EXPIRED
→ REVALIDATING
→ EXECUTED / CANCELLED / FAILED
```

每个状态转换必须可审计。

Telegram 审批必须具备：

- 唯一 Plan ID；
- 一次性 Nonce；
- 用户 ID 白名单；
- Chat ID 白名单；
- 短时有效期；
- 重复点击幂等；
- 一个计划只能批准一次；
- 一个交易对同时最多一份有效待审批计划；
- 无响应自动过期；
- Telegram 断线默认不交易；
- Bot Token 泄露时不能直接下单；
- 所有批准、拒绝、超时、重复点击和错误操作均写入审计日志。

Telegram Bot 不应持有 Bybit 交易密钥。

更安全的设计是：

> Telegram Bot 只产生签名审批事件，Freqtrade 在读取审批事件后执行。

### 4.4 批准不等于立即执行

用户点击“批准”后，必须重新执行完整校验：

1. 计划是否过期；
2. 当前价格是否仍在允许入场范围；
3. 当前价格偏离生成计划时的幅度是否超限；
4. 最新 RiskSnapshot 是否存在且未过期；
5. 当前是否为 `ENTRY_ALLOWED`；
6. 是否处于 `CLOSE_ONLY` 或 Kill Switch；
7. 当前余额是否足够；
8. 交易所最小下单额和精度是否满足；
9. 是否已有相同 Plan ID 的订单；
10. 是否已有同交易对冲突仓位或挂单；
11. 信号所依据的 K 线是否仍是最新完成 K 线；
12. API、数据库和审计组件是否处于健康状态。

任一检查失败，应取消执行并通知用户：

```text
计划未执行。
原因：当前价格已超出批准的入场范围。
```

### 4.5 LLM 不得控制最终风险预算

LLM 可以提出：

- 买入、持有或退出；
- 入场区间；
- 失效条件；
- 止损建议；
- 止盈方案；
- 最大持有期限；
- 决策理由；
- 风险提示。

LLM 不应直接决定：

- 最终仓位；
- 单笔风险比例；
- 每日亏损上限；
- 每周亏损上限；
- 最大回撤；
- Kill Switch；
- 是否解除 Close-Only；
- 是否绕过最小下单额和精度；
- 是否允许超出余额；
- 是否因高置信度扩大仓位。

最终仓位必须由现有确定性风险模块计算。

模型输出的 `confidence` 只能用于展示和研究，不得直接映射为仓位倍数。

### 4.6 增加模型输入边界

第一版建议只向 LLM 输入经过验证的结构化数据：

- 最近若干根已完成 K 线的摘要；
- Donchian、ATR、EMA、成交量等指标；
- 当前价格；
- 当前持仓；
- 账户 NAV；
- RiskSnapshot；
- 当前风险状态；
- 最近交易记录摘要；
- 现有挂单和待审批计划；
- 明确的市场规则。

第一版不建议直接输入：

- Twitter/X 自由文本；
- Telegram 群消息；
- 网页评论；
- 未清洗新闻正文；
- 用户自由文本交易指令；
- 可能包含 Prompt Injection 的外部内容。

如后续接入新闻和社交数据，应先经过独立的数据清洗、来源白名单和 Prompt Injection 防护。

## 5. AI 专用风险登记

| 风险 | 默认处理 |
|---|---|
| JSON 输出损坏 | 拒绝计划 |
| 字段缺失或未知字段 | 拒绝计划 |
| 模型虚构价格 | 以本地行情为准并拒绝计划 |
| 模型虚构持仓 | 以 Runtime DB 和 RiskSnapshot 为准 |
| 模型输出不支持交易对 | 拒绝计划 |
| 模型版本漂移 | 冻结并重新验证 |
| Prompt 变化未登记 | 禁止进入 Paper/Live |
| API 超时 | 不生成计划 |
| API 余额不足 | 不生成计划并告警 |
| 自动切换供应商 | 默认禁止 |
| Token 异常增长 | 触发预算告警和熔断 |
| Prompt Injection | 外部内容视为不可信数据 |
| Telegram 账号被盗 | 用户白名单、短 TTL、Nonce |
| 重复点击批准 | 幂等拒绝 |
| 计划审批疲劳 | 限制每日候选计划数量 |
| 价格审批期间漂移 | 执行前重新校验 |
| 风险快照过期 | Fail-Closed |
| 模型建议全仓 | 由风险模块强制裁剪或拒绝 |
| 模型建议取消止损 | 拒绝计划 |
| 模型输出过长 | Token 上限和输出长度限制 |
| 审计写入失败 | 禁止新入场 |

## 6. 验证与评估框架

### 6.1 必须区分 AI 与人工审批的贡献

人工批准会引入第二个决策者。最终表现不能只计算“系统赚没赚钱”，而应拆分为：

1. 所有 AI 候选计划的理论表现；
2. 用户批准计划的模拟或实际表现；
3. 用户拒绝计划的反事实表现；
4. 过期计划的反事实表现；
5. 被风险模块拒绝计划的反事实表现；
6. 原始 Donchian 基线表现；
7. AI 不参与时的传统策略表现；
8. AI 计划与基线计划的差异。

否则无法判断：

- AI 是否产生了价值；
- 用户是否提升了结果；
- 用户是否反而错过了优秀计划；
- 风险模块是否过滤了危险计划；
- 系统盈利来自模型、人工选择还是行情本身。

### 6.2 建议记录审批理由

用户拒绝计划时，建议选择固定理由：

- 风险过高；
- 理由不足；
- 行情已经变化；
- 与个人判断冲突；
- 已有同方向仓位；
- 不希望继续增加风险；
- 误操作；
- 其他。

用户批准后未执行，也应记录原因：

- 计划过期；
- 价格漂移；
- RiskSnapshot 失效；
- 余额不足；
- Kill Switch；
- 重复订单；
- 交易所拒单；
- 系统异常。

### 6.3 模型版本变更应重新开始验证

以下变化应视为策略实质变更：

- 更换模型供应商；
- 更换模型名称；
- 模型主版本升级；
- System Prompt 变化；
- 交易计划 Schema 变化；
- 输入指标变化；
- 决策字段变化；
- 审批有效期变化；
- 价格漂移阈值变化；
- 模型采样参数变化；
- 是否允许模型看到新闻或社交数据的变化。

发生上述变化时，建议：

1. 新建策略版本；
2. 更新 ADR；
3. 更新 Prompt Hash；
4. 重新运行离线测试；
5. 重启 Paper 计时；
6. 不继承旧版本的 Live 资格。

## 7. Final Holdout 改进意见

现有 Final Holdout 已被读取并降级，不应从已观察过的历史数据中重新切出一段并称为“未见数据”。

建议：

- 冻结唯一 AI 策略候选；
- 从冻结日期后积累前向未见数据；
- Shadow 模式记录所有 AI 计划；
- Paper 模式记录审批和模拟成交；
- 将该期间作为新的前向验证区间；
- 不在验证期间根据收益结果反复调整 Prompt；
- Prompt 或模型变化后重新开始计时。

同时应区分：

- 市场未见数据；
- 模型安全测试集；
- Prompt Injection 测试集；
- 极端行情测试集；
- Telegram 审批故障测试；
- 交易所异常测试。

---

## 8. 进度账本改进意见

开发计划顶部状态、任务正文状态、提交记录和下一可执行任务必须保持一致。

建议每次更新统一修改：

- 文档顶部当前阶段；
- 下一任务；
- 任务状态；
- 对应 Commit；
- 测试数量；
- 阻塞项；
- 延期项；
- 门禁结论；
- 评审人；
- 证据链接或 Hash。

禁止出现：

- 顶部显示 P3-03 未开始；
- 任务正文却显示 P3-03 已完成；
- 测试数量前后不一致；
- “项目所有人批准”被误写为独立评审；
- 已完成模块被重复开发。

---

## 9. 独立评审问题

当前计划要求开发者之外的独立评审者，但个人项目可能没有正式团队。

建议明确：

- 谁是实现者；
- 谁是项目所有人；
- 谁是风险批准人；
- 谁是独立评审者；
- 是否存在真实的第二人；
- AI 助手是否仅提供审查建议，而不冒充独立责任人。

如没有第二位真实评审者，应如实记录：

> 当前为单人项目，未满足真正独立评审条件。

不应以“项目所有人批准”替代独立评审。

## 10. 成本与复杂度控制

第一版应保留：

- Freqtrade 唯一交易写权限；
- Bybit API 禁止提现；
- IP 白名单；
- 确定性风险模块；
- Kill Switch；
- Telegram 审批；
- 审批幂等；
- Plan TTL；
- 执行前重新校验；
- 完整审计日志；
- Paper；
- 小额 Live Canary。

第一版暂不建议增加：

- 多模型辩论；
- 多 Agent 协作；
- 自动反思 Agent；
- 新闻 Agent；
- Twitter 情绪 Agent；
- 向量数据库；
- 多交易所；
- 多账户；
- 合约与杠杆；
- 自动 Prompt 优化；
- 自动模型切换；
- 复杂 Web 管理后台。

## 11. 建议的新增任务拆分

### P0-09 AI Proposal Authority

冻结模型、Prompt、输入输出、Token、失败和版本策略。

### P0-10 Telegram Approval Contract

冻结状态机、身份认证、Nonce、TTL、幂等和签名事件。

### P2-09 LLM Plan Generator

实现结构化输入、模型调用、JSON Schema 校验和审计记录。

### P2-10 Deterministic Plan Validator

实现市场规则、价格关系、风险限制和拒绝原因。

### P3-09 Approval Event Store

实现待审批计划、审批事件、过期和幂等状态。

### P3-10 Telegram Bot Adapter

实现只读展示、批准、拒绝和状态通知，不持有交易密钥。

### P3-11 Execution Revalidation

实现批准后的最新价格、风险、余额和重复订单检查。

### P3-12 AI Fault Injection

测试超时、乱码、幻觉、重复审批、Bot 断线和版本漂移。

### P4-09 AI Shadow/Paper Evaluation

记录全部 AI 计划、批准、拒绝和反事实表现。

### P5-09 Human-Approved Live Canary

仅以极小资金运行，并限制每日计划数和最大总风险。

## 12. 建议优先级

### P0：立即修改

- 修正 AI 不进入决策链的旧定义；
- 新增 AI Proposal Authority；
- 新增 Telegram Approval Contract；
- 修复进度账本不一致；
- 明确 Final Holdout 新方案；
- 明确独立评审现实情况。

### P1：第一版上线前必须完成

- JSON Schema；
- Prompt 版本与 Hash；
- 模型调用审计；
- Telegram 状态机；
- Plan TTL；
- Nonce；
- 用户白名单；
- 执行前重新校验；
- 幂等；
- Fail-Closed；
- AI 故障注入测试。

### P2：Paper 阶段完成

- 全部 AI 计划统计；
- 批准与拒绝反事实评估；
- 与 Donchian 基线对比；
- 模型成本和 Token 统计；
- 模型稳定性报告；
- 审批疲劳评估。

### P3：Live Canary 前完成

- 小额资金限制；
- 每日计划数量限制；
- 交易所拒单处理；
- 模型服务中断演练；
- Telegram 中断演练；
- API Key 轮换；
- 审计和恢复演练；
- 明确人工停机流程。

## 13. 最终审核结论

审核结论为：

> 有条件通过，需重大修改后重新评审。

原开发计划的传统量化工程基础、风险边界、审计设计和唯一交易写路径可以继续使用，但新增的 LLM 候选交易计划与 Telegram 人工审批机制必须正式纳入架构和门禁。

最重要的设计原则应保持为：

1. LLM 只生成候选交易计划；
2. LLM 不持有交易权限；
3. LLM 不控制最终风险预算；
4. Telegram 批准不直接等于下单；
5. 执行前必须重新校验；
6. Freqtrade 是唯一订单写入者；
7. 模型、Prompt 和 Schema 变化视为策略版本变化；
8. 模型失败、审批失败或审计失败时默认不交易；
9. Paper 必须区分 AI、人工审批和基线策略的贡献；
10. 未经新的 Paper 和 Live Canary 验证，不得将 AI 计划链用于真实资金。
