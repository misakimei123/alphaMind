# AI 辅助量化工程实践：外部经验评估与 alphaMind 融入建议

## 文档定位

本文评估 LINUX DO 帖子[《给了 GPT 200$，让它在 OKX 自由交易，为期两天，它能赚吗》](https://linux.do/t/topic/2629885/)及其中 RainsFly 的[回复 116](https://linux.do/t/topic/2629885/116)与[回复 128](https://linux.do/t/topic/2629885/128)。目标不是复刻帖子中的交易方式，也不把个人自述视为收益证明，而是筛选其中对 alphaMind 有工程价值的实践，并明确其适用边界。

本文是外部经验评估，不是新的规范性计划。任务状态、实施顺序和验收门禁仍以[完整开发计划](development-plan.md)为唯一依据；如本文与开发计划、ADR 或 Runtime Contract 冲突，以开发计划为准。

## 来源事实与证据边界

RainsFly 的两条回复可归纳为三项实践：

1. 从零使用 Codex 共同讨论策略，并生成量化框架与代码；
2. 将系统部署到服务器，持续运行 dry-run，由 Codex 协助排查日志和监控；
3. dry-run 观察近一个月后再尝试小额运行，同时把模型订阅费纳入“是否真正赚钱”的判断。

这些描述与 alphaMind 的方向有明显相似性，但证据强度有限：回复没有提供可复核的交易流水、净值曲线、策略版本、样本量、手续费与滑点明细、最大回撤、故障记录或对照实验。因此，可以借鉴其**工程路径和成本意识**，不能据此推导“LLM 交易有效”“dry-run 盈利可稳定复现”或“一个月足以验证策略”。

## 核心结论

最值得 alphaMind 融入的不是“让 GPT 自由交易”，而是以下闭环：

> AI 辅助研发与运维 → 版本冻结 → 持续 dry-run 收集证据 → 确定性门禁 → 小额 Live Canary → 对账与复盘 → 再决定是否扩展。

这个闭环与 alphaMind 当前架构同向，但必须保留 alphaMind 已冻结的三条安全边界：

- AI 只生成结构化建议，不持有交易所 key，不直接创建订单；
- Telegram 人工批准是普通交易动作进入执行链的授权点；
- Freqtrade 是唯一订单管理者和交易所写入者，风险、数量与杠杆由确定性程序裁决。

## 建议直接吸收的实践

### 1. 将 Codex 定位为研发与只读运维助手

Codex 很适合参与策略假设讨论、代码实现、配置审查、日志归因、故障摘要和改进建议。上线后也应继续利用这一能力，但输入应是经过脱敏和结构化整理的运行证据，输出只能是诊断或待审核的变更建议。

建议为后续运行阶段形成标准化的 `Incident Bundle`，至少包含：

- 事件时间、环境、构建版本、策略版本、prompt/model/provider 版本；
- DecisionContext 摘要、审批事件、ExecutionGateway intent 与 Freqtrade order/trade 标识；
- 重试、超时、拒绝、对账差异和 Kill Switch 状态；
- 对应时间窗的市场、仓位、PnL 与成本快照；
- 已执行的确定性保护动作和仍需人工决定的事项。

这样 Codex 可以高效分析问题，同时不会成为实时控制面或新的交易权限主体。

### 2. 把长时间 dry-run 当作运行可靠性门禁

持续 dry-run 的价值主要是暴露实时链路问题，而不是证明策略能赚钱。alphaMind 在 R4-05 的至少 7 天 dry-run 中，应持续记录并评估：

- 决策周期是否漏跑、重复运行或使用过期 DecisionContext；
- AI 输出是否稳定满足 schema，HOLD 与候选动作是否可解释；
- Telegram 审批 TTL、nonce、重复回调和拒绝路径是否正确；
- ExecutionGateway 是否产生重复 intent、悬空状态或越权写入；
- Freqtrade 模拟订单、重启恢复和自动对账是否一致；
- provider 超时、预算耗尽、新闻异常和行情缺失时是否 fail-closed；
- 每周期的模型、数据与基础设施成本是否可预测。

观察期达到日历天数不是晋升理由。只有关键故障指标、恢复证据和对账结果达到预先冻结的门禁，才允许进入下一阶段。

### 3. 同时报告交易表现与系统经济性

RainsFly 提到“总体盈利但未覆盖 GPT Pro 订阅费”，揭示了交易 PnL 与系统经济 PnL 不能混为一谈。alphaMind 已有模型调用用量与成本台账，后续应在 R4-R6 报告中并列呈现三套口径：

| 口径 | 建议定义 | 用途 |
|---|---|---|
| Execution PnL | 按实际成交计算的市场盈亏减交易手续费与 funding | 判断交易与执行链本身的效果 |
| Operating Cost | 模型调用、新闻/行情数据、服务器与监控等运行成本 | 判断维持系统所需的真实支出 |
| System Economic PnL | Execution PnL 减 Operating Cost | 判断整个系统是否具有经济价值 |

三者必须分账，不能把固定订阅费写入交易所账户 NAV，也不能只展示毛利润。实际成交价已经反映滑点，因此还应相对决策时基准价格单列 slippage 指标，但不得在 Execution PnL 中重复扣减。固定成本的摊销周期、共享资源的分摊规则和币种换算时点应版本化，以免事后改变口径。

### 4. 以小额 Live Canary 验证真实写路径

小额运行的合理目标不是追求收益，而是验证 dry-run 无法覆盖的真实交易所行为：认证、精度、最小下单量、成交、partial fill、手续费、撤单竞争、离线事件和对账。

alphaMind 的 Live Canary 应继续遵守当前 R6 边界：

- 现货先行，合约独立晋升；
- 冻结策略、prompt、model、provider、风险参数与部署版本；
- 使用专用、最小权限 key，AI 和 Telegram Bot 不可见；
- 资金和单笔风险由预先批准的硬上限约束；
- 扩大资金不是自动动作，必须基于独立复核后的新决策；
- 出现对账差异、孤儿订单、风险快照失效或保护链异常时立即 fail-closed。

## 需要改造后吸收的实践

| 外部实践 | 直接照搬的问题 | alphaMind 的吸收方式 |
|---|---|---|
| 与 AI 商量策略并让其写代码 | 容易把策略想法、实现正确性和收益证据混为一体 | AI 辅助提出与实现假设；假设、版本、证伪条件和晋升证据分别管理 |
| dry-run 有利润后小额运行 | 模拟盈利不能证明真实成交，也可能来自短样本或特定行情 | 同时检查可靠性、对账、风险、成本和样本覆盖；PnL 只是输入之一 |
| 让 Codex 定时检查日志和监控 | Codex 不应成为实时止损、告警或恢复所依赖的常驻组件 | 确定性 watchdog、指标和告警负责发现与保护；Codex 只做二次分析 |
| 给 AI 交易所 API key 后获取信息并下单 | 权限集中、提示注入和误操作可能直接作用于资金 | 读写分离；只读信息由受控适配层提供，唯一写路径保持为 ExecutionGateway → Freqtrade → Bybit |
| 根据近期表现继续调整 prompt | 容易形成追涨杀跌式人工过拟合，破坏评估可复现性 | 评估窗口内冻结 prompt/model/provider；变更进入新版本和新的验证周期 |

## 明确不应引入的做法

以下做法与 alphaMind 的安全和验证合同冲突，应明确拒绝：

1. 把主账户或可提现 API key 直接提供给 GPT、Codex、prompt 或聊天上下文；
2. 允许 AI、Telegram Bot 或临时脚本绕过 Freqtrade 直接向 Bybit 写单；
3. 把“两天或一个月有利润”当作策略有效性的充分证据；
4. 用 prompt 中的自然语言限制替代确定性的仓位、损失、杠杆和 Kill Switch；
5. 依赖 Codex 的定时唤醒承担实时监控、止损、对账或灾难恢复；
6. 在同一评估窗口内根据短期盈亏反复修改策略、prompt 或风险参数；
7. 只报告交易所余额变化，忽略手续费、funding、滑点、模型、数据和服务器成本。

## 与 R4-R6 的建议落点

| 阶段/任务 | 建议融入点 | 验收证据 |
|---|---|---|
| R4-01 ExecutionGateway POC | 保持单一写入者；为 intent、审批和 Freqtrade 事件建立稳定关联标识 | 重复请求幂等、过期审批拒绝、无旁路写单证据 |
| R4-04 恢复与自动对账 | 生成可供人工与 Codex 阅读的 Incident Bundle，但保护动作不依赖 AI | 重启恢复、孤儿订单检测、差异 fail-closed 与审计记录 |
| R4-05 Spot dry-run | 增加周期可靠性、审批链、对账、provider 成本和系统经济性报表 | 至少 7 天连续证据；晋升标准不以短期盈利单独决定 |
| R5 合约阶段 | 延续相同证据链，并单独验证 funding、杠杆、强平距离和 reduce-only | Spot 证据不替代 Futures 证据；异常时只减仓或停止 |
| R6 Live Canary | 冻结全套版本，以极小真实资金验证生产写路径和对账 | 真实成交、费用、滑点、恢复与 Kill Switch 证据；独立人工复核 |

上述落点是后续任务的验收补充建议，不代表相应能力已经实现。当前仓库尚未完成 R4 ExecutionGateway 和真实交易所写路径，因此不能把帖子中“已经部署并可下单”的成熟度类比为 alphaMind 的现状。

## 反方观点与失败路径

### 反方观点一：人工审批会错过行情，直接让 AI 下单更完整

这在高频或极短周期交易中成立，但 alphaMind 的目标周期默认是 30 分钟，且第一阶段资金规模有限。当前主要风险是错误动作、权限旁路和状态不一致，而不是微秒级延迟。人工审批带来的可追责性和资金保护高于其延迟成本。若未来需要无人值守策略，应作为具有独立风险预算和确定性规则的新系统评估，不能通过放宽 AI 权限渐进偷渡。

### 反方观点二：把订阅和服务器成本算入收益会扭曲策略评价

如果只研究交易策略的市场边际，这些固定成本确实不应混入 Execution PnL；但如果判断个人系统是否值得长期运行，忽略它们又会高估实际价值。因此正确做法是分账并列，而不是选择性只保留其中一个口径。

### 主要失败路径

- **运维自动化幻觉**：日志能被 AI 解读，不等于故障会被及时发现和安全处置；
- **短样本幸存者偏差**：一个月的盈利可能只覆盖单一市场状态，且未包含尾部风险；
- **持续调参污染证据**：看到亏损即修改 prompt，导致每段结果来自不同系统；
- **权限边界侵蚀**：为方便排障临时提供写权限，最终形成无法审计的旁路；
- **成本口径漂移**：盈利时忽略订阅费，亏损时又将所有固定成本计入，结论不可复核。

## 最终判断

RainsFly 的实践与 alphaMind 在“AI 辅助构建量化系统、先 dry-run、再小额运行、持续分析日志”上确有异曲同工之处。其最高价值是提供了一条个人开发者可参考的工程工作流样本，而不是证明 AI 可以稳定战胜市场。

alphaMind 应吸收这条工作流的**迭代速度、持续运行意识和全成本意识**，同时用现有的人工授权、确定性风控、单一写入者、版本冻结、分层验证与自动对账合同约束它。这样既保留 AI 在研发和诊断上的优势，也避免把 AI 的不确定性直接放大为资金风险。
