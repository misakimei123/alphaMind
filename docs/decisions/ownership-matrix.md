## 1. 角色矩阵

| 角色 | 核心职责 | 主责任人 | 替补/复核人 | 不得同时独占的权限 |
|---|---|---|---|---|
| 项目负责人 | 范围、排期、资源和阶段协调 | 项目所有人（用户） | Phase 3 前补充紧急联系人 | 不能让实施代理自行批准 Live Canary |
| 策略负责人 | 假设、实现、实验登记和证据报告 | Codex（实施） | 项目所有人（门禁复核） | Codex 不能批准自己的 Backtest Qualified 结论 |
| 数据负责人 | 数据来源、manifest、质量和 holdout 隔离 | Codex（实施） | 项目所有人（门禁复核） | Codex 不能自行更换 final holdout |
| 风险审批人 | 资金边界、RiskSnapshot、阈值和 Kill Switch | 项目所有人（用户） | Live Canary 前补充外部复核 | 不能把 Codex 建议视为资金授权 |
| 运维负责人 | 部署、密钥、告警、备份、恢复和停机 | 项目所有人（用户） | Phase 3 前补充紧急联系人 | 不能绕过风险审批恢复新入场 |
| 可选外部复核 | 密钥撤销、紧急处置和高风险扩容建议 | 未指定 | 扩资或高杠杆前按需安排 | 不替代项目所有人的资金责任 |
| 资金所有人 | 计划资金、最大损失和 Live Canary 授权 | 项目所有人（用户） | 不适用：个人项目 | 不得把未记录的口头意向视为上线批准 |

## 2. 个人项目的最低独立性要求

alphaMind 是个人项目，项目负责人、策略负责人、数据负责人和运维负责人可以由同一人承担，但必须满足：

- 普通代码任务由实现者按测试完成，项目所有人确认真实资金和风险参数；
- 资金所有人必须明确确认最大可接受绝对损失；
- 真实交易 key 的创建、权限和撤销必须有可复核记录；
- Codex 不持有 key，不批准真实资金或提高杠杆；
- 无法安排外部复核不阻塞 R0-R5；扩资或高杠杆前建议复核紧急处置流程。

## 3. 团队项目的职责分离

如果 alphaMind 是团队项目：

- 每个角色必须填写具体姓名或稳定标识；
- 风险审批人与策略负责人原则上分离；
- 运维负责人必须有替补；
- 告警升级路径必须包含时限和最终停机权；
- Live Canary 至少需要资金所有人、风险审批人和运维负责人共同确认。

## 4. 当前结论

个人项目的研究与实现阶段责任安排已经明确：

- Codex 是实施代理，不持有资金、密钥或上线批准权；
- 项目所有人负责范围、资金、风险、门禁和运维决策；
- Phase 3 前补充紧急联系人、告警渠道和响应时限；
- Live Canary 前补充外部 runbook 复核。

P0-01 已完成。项目级最大可接受损失采用 10% 比例与 45 USDT 固定上限中的更严格者；Codex 无资金和上线授权。
